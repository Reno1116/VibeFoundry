# VibeFoundry

用于设计和开发工作效率工具的个人仓库。每个工具只走两个阶段：先确认需求并在 Figma 完成交互，再实现代码并持续调优。

## 两步工作流

### 1. 需求与交互

目标是先把“做什么”确认清楚，不在这一阶段写正式代码。

1. 从 [`docs/工具设计模板.md`](docs/工具设计模板.md) 复制一份设计文档。
2. 与 AI 确认目标、用户、核心流程、范围和成功标准。
3. 在 Figma 输出关键页面、状态和交互，并把链接写入设计文档。
4. 人工确认需求和 Figma 交互后，将文档状态改为 `已确认`。

第一步唯一交付物：

```text
projects/<工具名>/design.md
```

### 2. 开发与调优

以已确认的 `design.md` 和 Figma 为准，在同一个项目目录内完成代码、测试和迭代。

1. 建立最小可运行版本，先打通核心流程。
2. 对照 Figma 实现界面和交互。
3. 运行真实场景验证，修复问题并调优体验。
4. 在项目 `README.md` 记录启动方式、当前能力和已知问题。

需求或交互发生实质变化时，先回到第一步更新 `design.md` 和 Figma，再继续开发。

## 目录结构

```text
VibeFoundry/
├─ README.md
├─ docs/
│  └─ 工具设计模板.md
└─ projects/
   └─ <工具名>/
      ├─ README.md       # 启动方式、当前能力、已知问题
      ├─ design.md       # 已确认需求与 Figma 交互链接
      ├─ src/            # 代码；按技术栈调整
      └─ tests/          # 测试；需要时创建
```

仓库不再按 `brief / spec / validate` 拆分文档，也不要求维护额外的流程记录。一个工具的资料和代码始终放在同一个 `projects/<工具名>/` 下。

## 开始一个新工具

可以直接把下面这段话发给 AI：

```text
请按 UTF-8 读取 D:\UGit\VibeFoundry\README.md 和
D:\UGit\VibeFoundry\docs\工具设计模板.md。

我们先执行第一步：确认需求并在 Figma 输出交互。
在我明确确认前，不要进入代码开发。
确认后，把设计文档放到 projects/<工具名>/design.md；
随后执行第二步：实现代码、验证并调优功能。
```

## 基本约定

- 中文文档统一使用 UTF-8 和 LF。
- 项目目录使用简短的英文小写名称，例如 `file-organizer`。
- 不提前创建用不到的目录；技术栈需要什么再创建什么。
- Figma 不追求完整视觉稿，但必须覆盖核心流程、空状态、加载状态和失败状态。
- 未经确认的设计不进入开发；小型修复可直接在第二步内完成。

## Git 提交

提交标题使用：

```text
[design][新增] 确认文件整理工具需求与交互
[dev][新增] 完成文件整理核心流程
[dev][调优] 优化批量处理反馈
[repo][整理] 简化仓库工作流
```
