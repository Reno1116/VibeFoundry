# 示例设计文档

## 基本信息

- 任务名称：阶段判断 CLI 示例
- 当前阶段：spec
- 关联 brief：`docs/briefs/sample-task/示例需求卡片.md`

## 目标与结果

- 目标：提供一个最小可运行工具，演示如何从仓库文档过渡到代码实现
- 预期产出：一个 `python main.py --goal ... --constraints ...` 可运行脚本
- 成功标准：脚本能根据输入信息给出 `brief`、`spec` 或 `build` 建议

## 方案概览

- 方案摘要：使用 Python 标准库 `argparse` 接收参数，根据必填信息齐全度输出当前建议阶段
- 为什么选这个方案：依赖最少，容易运行，足够承担示例职责
- 不采用的备选方案：Web UI；配置文件驱动；外部规则库

## 输入输出与接口

- 输入：`--goal`、`--has-scope`、`--has-constraints`、`--has-spec`
- 输出：标准输出中的阶段建议与一句原因
- 关键接口 / 文件 / 命令：`projects/demo-tool/main.py`，`python main.py --goal "..." --has-scope --has-constraints`

## 数据流与核心逻辑

- 主流程：解析参数 -> 判断 goal 是否存在 -> 判断范围与约束是否齐全 -> 判断是否已有 spec -> 输出阶段
- 关键状态：信息不足、可进入 spec、可进入 build
- 异常与失败场景：未提供 goal 时必须回到 brief；参数缺失时保持保守判断

## 实现边界

- 本次实现：命令行判断逻辑与 README 使用说明
- 暂不实现：自动读取 docs、交互式提问、多语言输出
- 兼容性注意事项：默认按 Python 3.10+ 运行

## 测试与验收

- 最小验证路径：运行三组命令，分别得到 `brief`、`spec`、`build`
- 需要的测试：命令行手工验证
- 验收条件：输出结果符合示例文档中定义的阶段流转

- 是否足够进入 build：是
- 仍待确认项：无
