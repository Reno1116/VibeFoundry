# VibeFoundry 日常操作手册

这份手册是 `VibeFoundry` 的日常主入口，按你每天真实使用仓库的顺序来写。

如果你只想知道规则，请看 `docs/workflow-v1.md`。
如果你要马上开始一个任务，先看这份手册。

## 0. 每次开始前先做什么

1. 先确认当前任务是不是一个新任务
2. 新开和 AI 的对话时，先发送 `ai-kit/quick_start_prompt.md`
3. 明确告诉 AI：先按 `UTF-8` 读取所有中文文档
4. 不要一上来直接写代码，先判断当前该停在 `brief`、进入 `spec`，还是已经可以 `build`

常用命令：

```powershell
git status
git branch
git config commit.template .gitmessage.txt
```

## 1. 开一个新任务

先读：

- `ai-kit/quick_start_prompt.md`
- `ai-kit/README.md`

然后做：

1. 用 `quick_start_prompt.md` 开新对话
2. 让 AI 判断当前信息是否足够进入 `spec`
3. 如果信息不够，就先写 `brief`

产出放这里：

- `docs/briefs/任务名/`

什么时候不能进入下一阶段：

- 目标不明确
- 范围不明确
- 约束不明确
- 成功标准不明确

什么时候提交 git：

- `brief` 第一次定稿后提一次

建议提交：

```text
[brief][新增] 补齐任务目标、范围与成功标准
```

## 2. 写 brief

先读：

- `docs/templates/需求卡片模板.md`

然后做：

1. 补齐目标用户、核心问题、成功标准
2. 写清本次要做和不做什么
3. 写清时间、技术、资源、外部依赖约束
4. 在“进入下一阶段的条件”里检查四要素是否齐全

产出放这里：

- `docs/briefs/任务名/`

什么时候不能进入下一阶段：

- 四要素检查里有任一项还不明确

什么时候提交 git：

- `brief` 达到可进入 `spec` 的状态时提交

常用命令：

```powershell
git checkout -b brief/任务名
git add docs/briefs
git commit
```

## 3. 进入 spec

先读：

- `docs/templates/设计文档模板.md`
- 对应任务的 `brief`

然后做：

1. 把方案写到“可直接开干”的深度
2. 明确输入输出
3. 明确边界和失败场景
4. 明确最小验证路径和验收条件

产出放这里：

- `docs/specs/任务名/`

什么时候不能进入下一阶段：

- 输入输出没写清
- 边界或失败场景没写清
- 验证方式没写清

什么时候提交 git：

- `spec` 达到“可直接开干”时提交

建议提交：

```text
[spec][修改] 明确输入输出与 build gate
```

## 4. 进入 build

先读：

- 对应任务的 `brief`
- 对应任务的 `spec`

然后做：

1. 先做一次人工确认，确认现在真的可以开始实现
2. 只有确认通过，才创建 `projects/任务名/`
3. 先做最小可运行版本，不追求一步做全
4. 优先打通主流程，再补细节

产出放这里：

- `projects/任务名/`

什么时候不能进入下一阶段：

- 还没人工确认
- `spec` 还存在关键待确认项

什么时候提交 git：

- 最小可运行版本已经跑通时提交

常用命令：

```powershell
git checkout -b build/任务名
git add projects/任务名
git commit
```

建议提交：

```text
[build][修复] 修正命令行工具 UTF-8 输出
```

## 5. 补 validate

先读：

- `docs/templates/验收模板.md`
- 对应任务的 `spec`
- 本轮实现结果

然后做：

1. 写清是否完成预期目标
2. 写清本轮怎么验证的
3. 写清未覆盖项、已知问题和后续建议
4. 明确任务是继续迭代还是归档

产出放这里：

- `docs/validate/任务名/`

什么时候不能进入下一阶段：

- 没写验证方式
- 没写缺口
- 没写下一步结论

什么时候提交 git：

- 最小验收记录补完后提交

建议提交：

```text
[validate][整理] 补充最小验收记录与已知缺口
```

## 6. 继续迭代还是归档

继续迭代：

- 任务还有下一轮明确目标
- 当前产出仍是活跃资产

归档：

- 暂停
- 冻结
- 废弃

产出放这里：

- 继续迭代：保留在当前活跃目录
- 归档：移入 `docs/archive/`

建议提交：

```text
[repo][归档] 归档已暂停任务的 brief、spec 与 validate
```

## 7. 如果你一开始就想写代码

默认流程不是直接开始实现，而是：

1. 先补最小 `brief`
2. 至少补齐目标、范围、约束、成功标准
3. 如果已经足够清楚，再进入 `spec`
4. 通过人工确认后再进入 `build`

只有在上下文已经非常完整时，才允许快速进入实现。

## 8. Git 提交规范

提交标题固定格式：

```text
[阶段][动作] 具体结果
```

阶段固定为：

- `brief`
- `spec`
- `build`
- `validate`
- `repo`

动作固定为：

- `新增`
- `修改`
- `修复`
- `重构`
- `整理`
- `归档`
- `删除`

摘要怎么写：

- 只写这次真正交付了什么
- 不写空泛词，例如“更新一下”“改点内容”
- 不用文件名列表代替结果

混合改动怎么定：

- 默认按主要交付物选择主阶段
- 不做多阶段并列标题
- 如果只是仓库级整理，使用 `repo`

提交正文固定写：

- 改动内容
- 影响范围
- 关联文档
- 后续事项

示例：

```text
[brief][新增] 补齐任务目标、范围与成功标准
[spec][修改] 明确输入输出与 build gate
[build][修复] 修正命令行工具 UTF-8 输出
[validate][整理] 补充最小验收记录与已知缺口
[repo][整理] 更新日常操作手册与入口说明
```

## 9. 你每天最常用的一套最小流程

```text
1. 发 quick_start_prompt
2. 判断先写 brief 还是直接进 spec
3. brief 定稿后提交一次
4. spec 写到可直接开干后提交一次
5. 人工确认后进入 build
6. 做出最小可运行版本后提交一次
7. 补 validate 后再提交一次
8. 判断继续迭代还是归档
```
