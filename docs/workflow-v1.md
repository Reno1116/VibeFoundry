# VibeFoundry Workflow v1

## 核心原则

- 新任务固定从 `brief` 开始
- AI 新对话固定先走 `ai-kit/quick_start_prompt.md`
- 中文文件读取前先声明按 `UTF-8` 解码
- 没有需求卡片的任务，默认不直接开始代码实现

## 主链路

1. `brief`
   先明确目标、范围、约束、成功标准。
2. `spec`
   只有四要素齐全后才进入，且文档要写到可直接开干。
3. `build`
   进入实现前做一次人工确认；实现时优先最小可运行版本。
4. `validate`
   每次实现后都补最小验收记录，至少说明结果、验证方式、缺口。
5. `archive / continue`
   仍会继续迭代的任务保留在活跃区；暂停、冻结、废弃的任务归档。

## 阶段 gate

### Brief -> Spec

必须同时满足：

- 目标明确
- 范围明确
- 约束明确
- 成功标准明确

### Spec -> Build

必须同时满足：

- 输入输出已说明
- 核心边界已说明
- 验证方式已说明
- 经过一次人工确认

## 快速通道

如果用户直接要求写代码，但上下文不足：

- 先补最小 `brief`
- 不直接跳过文档阶段
- 补齐关键条件后再进入 `spec` 或 `build`

## Git 节奏

- 默认按阶段提交：`brief`、`spec`、`build`、`validate`
- 建议分支命名：`brief/...`、`spec/...`、`build/...`
- `projects/任务名/` 只在正式进入 `build` 时创建
- 提交标题固定格式：`[阶段][动作] 具体结果`
- 动作固定使用：`新增 / 修改 / 修复 / 重构 / 整理 / 归档 / 删除`
- 混合改动默认按主要交付物选择主阶段
